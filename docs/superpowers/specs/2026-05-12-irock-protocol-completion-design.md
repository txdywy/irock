# Protocol Completion Local Wire-Format Design

Date: 2026-05-12

## Goal

Complete the previously planned Alpha protocol set in the repository's local test boundary by replacing protocol foundation marker payloads with deterministic, testable protocol wire-format builders and by extending tests until every planned protocol path is covered by `swift test`.

The acceptance boundary is local repository verification. Real external node interoperability remains a manual operator validation activity because it requires private credentials, reachable servers, network conditions, and platform signing state that must not be committed to the repository.

## Planned protocol surface

The planned Alpha surface remains:

- Shadowsocks over TCP/TLS-supported transport selection.
- VMess over TCP, TLS, WebSocket, HTTP/2, and gRPC transport paths.
- VLESS over TCP, TLS, WebSocket, HTTP/2, gRPC, plus the Reality transport path.
- Trojan over TCP, TLS, WebSocket, HTTP/2, and gRPC transport paths.
- Hysteria2 over QUIC.
- TUIC over QUIC.

## Current gap

Shadowsocks already has real local stream framing and AEAD/SS2022 codec coverage. VMess, VLESS, Trojan, Hysteria2, TUIC, Reality, WebSocket, HTTP/2, gRPC, and QUIC currently use credential-safe foundation marker payloads. These markers are useful seams, but they are not protocol-shaped wire bytes and they make it too easy to mistake adapter routing coverage for protocol completion.

## Scope

- Keep the existing `ProxyAdapter` to `TransportRequest` boundary.
- Replace protocol marker payloads with protocol-specific binary builders where the repository can verify the wire format locally.
- Add dedicated request/framing builders for Trojan, VLESS, VMess, Hysteria2, and TUIC.
- Upgrade transport foundation markers into deterministic wire-format builders for WebSocket, HTTP/2, gRPC, QUIC, and Reality metadata/preamble where possible without platform-specific networking.
- Expand tests so each planned protocol and transport combination has local coverage for validation, metadata redaction, transport selection, and generated bytes.
- Keep all credentials, node URIs, keys, salts, and real server material out of committed fixtures.

## Out of scope

- Automated tests against real external nodes.
- Embedding sing-box, xray, clash, or any full proxy core.
- Shipping committed credentials, subscription URLs, private keys, or real node captures.
- Full HTTP/2 HPACK implementation, complete QUIC stack, uTLS impersonation, or full Reality cryptographic handshake in this phase.
- Packet Tunnel entitlement/signing validation.
- UDP relay completeness beyond existing local foundation coverage.

## Architecture

### Protocol request builders

Each protocol gets a small focused builder that owns its first payload bytes and validation rules. The builders remain independent of sockets and return deterministic `Data` plus credential-safe metadata.

- `TrojanOpenRequest` builds the Trojan request prefix: SHA224 password hex, CRLF, command byte, address frame, port, and CRLF.
- `VLESSOpenRequest` builds a VLESS request header with version, UUID bytes, add-on length, command, port, and address encoding. The local scope accepts `none` encryption and validates unsupported flow/security combinations explicitly.
- `VMessOpenRequest` builds a deterministic VMess-shaped request header for the currently configured TCP/TLS transport scope, including UUID validation, command, port/address encoding, security selection, and authenticated-header test seams. VMess cryptographic fields that require wall-clock or random material must accept injectable deterministic values in tests.
- `Hysteria2OpenRequest` builds a QUIC-path local handshake/auth descriptor in binary form instead of a plaintext marker, including authentication presence, SNI, and obfuscation metadata without exposing secrets.
- `TUICOpenRequest` builds a QUIC-path local authentication descriptor in binary form, validating UUID/password format and keeping the password out of metadata and logs.

### Shared address encoding

Introduce or reuse one shared protocol address encoder for domain, IPv4, IPv6, and port fields. Trojan, VLESS, VMess, and Shadowsocks should use consistent destination validation while preserving protocol-specific address type bytes.

### Transport builders

Transport adapters continue to return `TransportConnection` through the existing dialer seams, but their `initialPayload` should become protocol-shaped:

- WebSocket builds an RFC6455 client upgrade request with deterministic test key injection and validates path/host metadata.
- HTTP/2 builds a client preface plus minimal deterministic SETTINGS/HEADERS representation suitable for local tests, while explicitly documenting that full HPACK and stream lifecycle are outside this phase.
- gRPC builds on the HTTP/2 local representation and adds deterministic gRPC message framing metadata for the selected service path.
- QUIC builds a binary local transport prelude carrying server name, ALPN, and protocol identifier rather than a plaintext marker.
- Reality builds a deterministic local client-hello descriptor from public key, short ID, server name, fingerprint, and spiderX metadata without attempting full X25519/uTLS interop.

### Adapter flow

The existing adapters keep their public shape:

1. Validate node protocol, host, port, credential reference, transport support, and protocol-specific options.
2. Resolve credentials through the credential resolver.
3. Build the protocol open request bytes.
4. Pass those bytes as `TransportRequest.initialPayload` to the chosen transport adapter.
5. Return a credential-safe `ProxyConnection` with generated metadata.

This limits churn: app, runtime snapshot, and tunnel code keep using the same adapter registry while protocol completeness improves behind the existing boundary.

## Error behavior

- Invalid credentials fail before network I/O.
- Unsupported protocol/security/flow/transport combinations fail with explicit `ProxyProtocolError.invalidConfiguration(...)` messages.
- Generated metadata records presence and selected modes, never raw passwords, UUID secrets beyond safe presence flags, full URIs, salts, keys, or payload bytes.
- Transport builders reject invalid paths, hosts, ALPN lists, or Reality options before invoking the underlying dialer.
- Tests must include secret-redaction regressions for every protocol error path that touches credential-derived values.

## Testing strategy

- Add unit tests for shared address encoding across domain, IPv4, IPv6, and invalid destinations.
- Add Trojan request vector tests for SHA224 password hex, CRLF delimiters, command byte, address, and port.
- Add VLESS request vector tests for UUID bytes, version, add-on length, command, address, port, and unsupported option rejection.
- Add VMess request vector tests with deterministic random/time injection and security selection coverage.
- Add Hysteria2 and TUIC binary local handshake descriptor tests for validation, metadata redaction, and QUIC adapter routing.
- Add WebSocket upgrade request tests with deterministic `Sec-WebSocket-Key` injection.
- Add HTTP/2, gRPC, QUIC, and Reality local wire-format tests that replace plaintext foundation-marker assertions.
- Expand adapter tests for every planned protocol and transport combination.
- Keep URI import tests focused on supported local parser boundaries and make unsupported parser gaps explicit instead of silently treating classification as full import support.
- Run full `swift test` after each implementation slice and before merge.

## Implementation sequence

1. Remove protocol marker payloads from Trojan by implementing and testing the real Trojan request prefix.
2. Implement and test VLESS request header bytes.
3. Implement and test VMess deterministic local request header bytes.
4. Replace Hysteria2 and TUIC plaintext marker payloads with binary local handshake descriptors and tests.
5. Replace WebSocket foundation payload with an RFC6455 upgrade request builder and tests.
6. Replace HTTP/2 and gRPC foundation payloads with local binary wire-format builders and tests.
7. Replace QUIC and Reality plaintext markers with binary local descriptors and tests.
8. Expand cross-protocol adapter matrix tests and run full verification.
9. Commit, push, merge, and clean local git state after verification succeeds.

## Acceptance criteria

- No planned protocol opener emits `*-foundation` plaintext marker payloads.
- All planned protocol and transport combinations are covered by local SwiftPM tests.
- `swift test` passes.
- Tests prove credentials are not exposed in metadata, errors, or committed fixtures.
- Design remains within the local verification boundary and does not require private real-node material.
- Existing Shadowsocks and macOS local proxy tests continue to pass.
